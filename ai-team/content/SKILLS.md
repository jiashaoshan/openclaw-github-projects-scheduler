## 任务执行规范

### 任务类型判断

执行任务前，先判断任务类型：

**GitHub Projects 任务**（任务来源: GitHub Projects）：
- 任务ID格式：`PVTI_xxx`
- **必须**自己更新GitHub状态

**直接对话派发任务**（任务来源: 直接对话派发）：
- 任务ID格式：普通字符串或对话ID
- **不需要**更新GitHub状态
- 直接群里汇报即可

### GitHub Projects 任务状态自更新

当执行 GitHub Projects 自动任务时，子Agent必须自己更新任务状态：

**执行成功时**：
```python
import subprocess
# 标记任务完成
subprocess.run([
    "python3", 
    "~/.openclaw/workspace/skills/github-projects/task_scheduler.py",
    "--complete", 
    "任务ID"  # 从任务描述中获取
])
```

**执行失败时**：
```python
import subprocess
# 标记任务失败
subprocess.run([
    "python3",
    "~/.openclaw/workspace/skills/github-projects/task_scheduler.py", 
    "--fail",
    "任务ID:失败原因"
])
```

**重要**：
- 先判断任务类型（看"任务来源"字段）
- 只有GitHub Projects任务需要更新状态
- 直接对话任务只需群里汇报
- 完成后立即处理，不要等待主Agent

## 行为准则（Karpathy 原则适配版）

> 源自 Andrej Karpathy 的 LLM 最佳实践，用于减少内容创作中的常见错误。

### 1. 写作前思考
**不要假设。不要隐藏困惑。呈现权衡。**

- 明确说明对主题的理解，如果不确定，询问而不是猜测
- 当存在多种风格/角度选择时，呈现它们，不要默默决定
- 如果存在更简洁的表达方式，说出来，适时提出异议
- 对模糊需求停下来，指出不清楚的地方并要求澄清

### 2. 简洁优先
**用最精炼的内容传达核心信息。不要过度包装。**

- 不添加要求之外的冗余内容
- 不为一次性文案创建复杂模板
- 不堆砌不必要的修辞和华丽辞藻
- **如果 500 字能说清楚，不写 1000 字**

**检验标准：** 读者会觉得这过于啰嗦吗？如果是，精简。

### 3. 精准修改
**只改必须改的。不动未要求的部分。**

- 不"优化"用户提供的原始素材（除非明确要求）
- 不擅自改变既定风格/语气
- 匹配用户指定的格式要求
- 如果注意到更好的表达方式，提建议但不直接替换

**检验标准：** 每一处修改都应该能直接追溯到用户的明确要求。

### 4. 目标驱动交付
**定义成功标准。验证直到达成。**

将模糊需求转化为可验证目标：
- "写篇爆款" → "标题点击率>5%，正文完读率>30%"
- "专业一点" → "使用行业术语，结构清晰，逻辑严密"
- "活泼一些" → "多用emoji，短句为主，语气轻松"

交付前自检：
```
□ 是否满足明确的风格要求？
□ 是否在指定字数范围内？
□ 是否包含要求的元素（关键词、hashtag等）？
□ 是否符合平台调性？
```

**关键：** 强有力的成功标准让你能独立交付高质量内容。模糊目标（"写得好一点"）需要不断返工。
